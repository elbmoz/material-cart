# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

STM32F407VETx competition robot (工创) — 4-wheel mecanum chassis with a 2-DOF servo arm and Z-axis lift. Generated by STM32CubeMX 6.8.0, built with Keil MDK-ARM.

## Build / Flash

This is a Keil MDK project. No CLI build tooling is configured. The project file is `MDK-ARM/gc.uvprojx`. Open it in Keil uVision to compile and flash via ST-Link (SWD on PA13/PA14).

Preprocessor defines: `USE_HAL_DRIVER;STM32F407xx`

## Architecture

### Timing & Control Loop

TIM2 generates a 1ms interrupt that increments `pit_1ms_pending` and `system_count` (declared in `GC_Chassis_Control.c`). Every 20ms, `pit_control()` runs all **five** PID controllers at 50Hz. Each controller checks its own enable flag and returns immediately if disabled. PID sample time `PID_DT = 0.02f` matches the 20ms call period.

**Note:** In the current revision, `pit_control()` is commented out of `main()`'s `while(1)`. Controllers are instead driven via blocking wrappers (`*_Block()` functions, `Mission_PitDelay()`) that call `pit_control()` internally.

### Module Layers (bottom-up)

1. **zhangdatou.c** — Low-level motor driver protocol. Sends commands to stepper motor drivers over USART1 (115200 baud, RS485). Handles enable/disable, speed mode, position mode, sync start, and position/speed readback from all 5 motors (4 chassis + Z-axis).

2. **Motor_Move.c** — Chassis kinematics and Z-axis control. Wraps `zhangdatou.c` with mecanum-wheel directions for straight/back/left/right/turn movements (both speed and position modes). Also provides `DifferentialTurn()` for arc turns, `Stop_car()`, and Z-axis homing, relative/absolute moves, limit switch detection (`PE7`, active-low), and `ZAxis_Home()` with timeout. Motor addresses: LF=0x01, RF=0x02, LB=0x03, RB=0x04, Z_AXIS=0x05.

3. **PID.c** — Pure PID algorithm library. Provides `PID_Cycle` struct (p/i/d gains, limits, deadband, mode, runtime state), `PID_Cycle_Reset()`, `PID_Cycle_SetEnable()`, and `PID_Control()`. Also exports utility functions `LimitValue()` and `Abs()`. Supports both position-form and incremental-form PID via the `PID_POSITION` / `PID_INCREMENTAL` mode flags.

4. **GC_Chassis_Control.c** — Chassis PID orchestration layer. Contains:
   - **Four** cascade structs in the header: `CarStraight_Cascade` (outer position + inner yaw PID), `InPlaceTurn_Cascade` (single yaw PID), `InPlaceTurnMm_Cascade` (yaw PID with minimum speed enforcement — prevents stall at small errors), `VisionPrecision_Cascade` (independent x/y/z PIDs fed by camera data)
   - `PID_Config_Init()` — one-time initialization of all controller gains, limits, and deadbands (idempotent via `pid_config_done` flag)
   - `*_Start()` / `*_Control()` / `*_Stop()` functions for each controller. Start functions mutually disable the other controllers. Control functions implement mecanum wheel kinematics via `ChassisSetWheels()`
   - `VisionPrecision_Update()` — called from ISR context to feed camera x/y error data; uses `volatile` members for ISR-sharing (`x_error`, `y_error`, `data_valid`, `data_timeout_count`)
   - `pit_control()` — the 1ms PIT consumer loop: runs `InPlaceTurn_Control()`, `InPlaceTurnMm_Control()`, `CarStraight_Control()`, `VisionPrecision_Control()`, and `ServoMotorVision_Control()` at 50Hz. Also sends Bluetooth yaw telemetry at ~2Hz
   - `Mission_PitDelay(ms)` — blocking delay that keeps calling `pit_control()` so PID loops stay alive during waits
   - Blocking convenience wrappers: `InPlaceTurn_Block()`, `InPlaceTurnMm_Block()`, `CarStraight_Block()`

5. **HWT101.c** — IMU driver on USART2 (115200 baud). Outputs `global_angle` (yaw as double), `angular_velocity_y`, `angular_velocity_z`. Initialization sends a zeroing command (`guiling`) and starts interrupt reception. Parses 11-byte angle (0x53) and angular velocity (0x52) packets via `hwt_Handler()` with checksum validation.

6. **serial.c** — Vision module communication on UART5 (9600 baud). Receives lines formatted as `"x_error,y_error\n"` or `"none\n"` (for no detection), parses them, and feeds data to **both** `VisionPrecision_Update()` and `ServoMotorVision_Update()` simultaneously — the chassis and servo arm both consume the same vision error stream. Stream start/stop commands: `'c3'` to start, `'ok'` to stop. Exports `Serial_Printf()` and `fputc()` for printf-over-serial.

7. **bluetooth.c** — Debug telemetry, shares UART5 (9600 baud, TX-only). `Bluetooth_SendAngle()` sends yaw at ~2Hz from `pit_control()`. `Bluetooth_SendPos()` sends position diagnostics (current, target, diff). Uses blocking `HAL_UART_Transmit()`.

8. **LobotServoController.c** — LSC-series serial servo driver on USART6 (9600 baud). Communicates with Lobot serial bus servos using 0x55-framed protocol:
   - Servo 0: 180° position servo (arm rotation)
   - Servo 12: claw/gripper
   - Servo 13: 360° continuous rotation servo (Y-axis fine positioning)
   - Key functions: `moveServo(ID, Position, Time)`, `moveServos()`, `servo13_Home(backOffMs)` — homes servo 13 by forward-spinning until PE7 limit switch, then reverses by `backOffMs`. `servo13_SetSpeed(speed)` — -1000 to +1000 continuous speed control (PWM=1500+speed, Time=0 for continuous)
   - Uses `HAL_UARTEx_ReceiveToIdle_IT` for RX

9. **huaner_servo.c** — Servo preset positions (thin wrapper over LobotServoController):
   - `kan()`: servo 0 → 2390 (arm to look-forward position)
   - `zhua()`: servo 12 → 1365 (claw grab)
   - `fang()`: servo 12 → 1105 (claw release)
   - `fangche1/2/3()`: servo 0 → 1220/965/680 (three platform-place heights)

10. **action.c** — High-level competition action sequences. `zhuafang1/2/3()` each perform: release claw → Z-axis down to grab height → grab → Z-axis up → move arm to platform height → release → arm back to home. Each variant targets a different platform height.

11. **servo.c** — PWM servo control on PA6/PA7 using TIM3 hardware PWM (50Hz, 500–2500µs). Provides `Servo_SetAngle()`, `Servo_SetBothAngle()`, `Servo_SetPulseUs()`. Still initialized in `main()` but the primary arm/gripper control has moved to the Lobot serial servo system (module 8).

12. **ServoMotorControl.c** — Second vision PID cascade for fine servo alignment. Contains `ServoMotorVision_Cascade` with independent X/Y PIDs:
    - X-axis: servo 0 (180° position servo), PID output = position offset from center (limited to ±1500 pulse units)
    - Y-axis: servo 13 (360° continuous rotation), PID output = speed offset from stop (limited to ±1000)
    - Same ISR-safe `volatile` pattern as chassis `VisionPrecision_Cascade`
    - `ServoMotorVision_Calibrate()` — one-shot blocking calibration: starts vision stream, runs PID loop until convergence, stops stream
    - Includes rate limiting on servo 0 commands (50ms minimum interval between `moveServo` calls)
    - Converges when X/Y errors are within deadbands AND Y-axis speed is near zero

13. **encoder_f407.c** — TIM4 encoder driver with PD13 direction pin. `Encoder_task()` reads counter increment, applies direction sign, and accumulates into `total_count`. `get_encoder_point()` returns read-only pointer to the encoder state struct. Designed to be called at 1ms intervals.

14. **QRcode.c** — QR code reader on UART5. Expects a 2-digit string like `"12"` (digits 1–3 for red/green/blue): first digit = pick color, second = place color. Sets `qrcode_task_ready` when a valid pair is received. Parsed character-by-character in UART RX interrupt. **Currently commented out** in `main()` init.

15. **delay.c** — DWT-based microsecond and millisecond delay (uses the Cortex-M4 cycle counter).

### USART Assignments

| Peripheral | Baud | Purpose |
|---|---|---|
| USART1 | 115200 | Stepper motor drivers (RS485 protocol) |
| USART2 | 115200 | HWT101 IMU |
| USART3 | 115200 | Configured, currently unused |
| UART4 | 9600 | Configured, currently unused |
| UART5 | 9600 | **Shared**: Vision/OpenMV (RX + stream commands) + Bluetooth debug (TX-only) |
| USART6 | 9600 | Lobot serial servo controller (arm, claw, continuous-rotation servos) |

### Startup Sequence (main.c)

1. Clock config (HSI), GPIO, all USARTs, TIM2 init
2. 200ms delay, DWT init, motor init + enable
3. Serial (vision) init on UART5, Bluetooth init on UART5
4. HWT101 IMU init on USART2
5. PWM servo init (TIM3, PA6/PA7) + Lobot serial servo init (USART6)
6. `servo13_Home(80)` — home continuous-rotation servo 13 against PE7 limit switch
7. `ZAxis_Home(1000, 200, 5000)` — home Z-axis upward until limit switch
8. `ZAxis_MoveRelative(-500, 200, 1000)` — short downward move
9. Start TIM2 interrupt (1ms PIT)
10. `kan()` — arm to look-forward position
11. `ServoMotorVision_Calibrate()` — one-shot vision-guided servo alignment (blocks until convergence)
12. `zhuafang3()` — grab and place action sequence
13. Enter empty `while(1)` — `pit_control()` is commented out; all work is done in startup

### Key Design Patterns

- **Mutual exclusion between chassis controllers**: Each `*_Start()` function disables the other three chassis controllers before enabling itself, ensuring only one chassis control mode is active at a time.
- **Dual-consumer vision data**: The same parsed `x_error,y_error` from `serial.c` feeds both `VisionPrecision_Cascade` (chassis movement) and `ServoMotorVision_Cascade` (servo fine-alignment) simultaneously. Both update from the same ISR callback.
- **Two independent motor buses**: USART1 (RS485, stepper motors) and USART6 (Lobot serial protocol, servos). Each has its own protocol, framing, and busy-check pattern.
- **Motor communication is sequential**: Commands are sent one motor at a time with 1ms delays between them, followed by `Motor_SyncStart()`. `Motor_IsComBusy()` guards against overlapping transactions.
- **PIT-based delays**: `Mission_PitDelay(ms)` is used instead of `HAL_Delay()` because the PID controllers must keep running during waits. It continually calls `pit_control()` while blocking.
- **ISR-safe data sharing**: Vision error data (`x_error`, `y_error`, `data_valid`) uses `volatile` qualifiers and is written from ISR context via `*_Update()` functions, read from main-loop context in the respective `*_Control()` functions.
- **One-time init guard**: `PID_Config_Init()` and `ServoMotorVision_ConfigInit()` use static flags — safe to call from multiple `*_Start()` entry points.
- **Stability debounce**: All controllers use a `stable_count` counter that increments when errors are within deadbands and resets otherwise. The controller declares `finished` only after N consecutive in-deadband samples.
- **Minimum speed enforcement**: `InPlaceTurnMm_Cascade` enforces a `min_turn_speed` — when PID output is below minimum but error is still outside deadband, it clamps to the minimum. Prevents stall at small errors.
- **Blocking wrappers**: `*_Block()` functions start a controller and busy-loop calling `pit_control()` until `finished` is set. Used for choreographed sequences where each action must complete before the next begins.
